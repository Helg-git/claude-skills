# Tech Competitive Analysis - 软件产品竞品分析

## Overview

软件产品竞品分析 = 六层拆解 + 混合信息收集 + 差异化机会识别

**核心原则**：不仅对比功能，更要理解竞品的架构决策和权衡，找到可执行的差异化切入点。

## When to Use

```
需求分析 → 需要了解市场现状？
              ↓
         竞品分析？
           ↙      ↘
        是         否
         ↓          ↓
    使用本技能    直接设计
```

**触发条件：**
- 软件/SaaS/DApp 产品开发前的市场调研
- 寻找产品差异化切入点
- 投资决策前的技术尽职调查
- 竞品功能追踪和对比

**不需要：**
- 纯技术选型（无商业对比需求）
- 已有明确方案仅需实现

## Analysis Depth Modes

```
┌─────────────────────────────────────────────────────────┐
│              选择分析深度                                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  scan（快速扫描）                                        │
│  ├─ 3-5 个主要竞品                                       │
│  ├─ 核心功能对比                                         │
│  ├─ 快速机会识别                                         │
│  └─ 适合：早期探索、快速决策                             │
│                                                         │
│  deep（深度分析，默认）                                   │
│  ├─ 5-10 个竞品（直接+间接+新兴）                         │
│  ├─ 完整六层拆解                                         │
│  ├─ SWOT + 可视化 + 技术选型                              │
│  └─ 适合：重要决策、投资分析、战略规划                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**默认使用 `deep` 模式，除非明确指定 `scan`。**

## Information Gathering（混合信息收集）

```
信息来源优先级：
├─ 1. 用户提供的竞品列表和补充信息
├─ 2. AI 内部知识（知名产品）
└─ 3. 联网搜索（最新动态、数据更新）
```

**搜索策略：**
- 使用 WebSearch 查找最新市场数据
- 交叉验证多个来源
- 优先使用权威来源（官方文档、知名分析机构）

## Six-Layer Analysis Framework（六层拆解法）

### 第一层：竞品分类

```markdown
| 类型 | 定义 | 示例 |
|------|------|------|
| 直接竞品 | 相同产品形态，争夺同一用户群 | |
| 间接竞品 | 不同形态但解决相同需求 | |
| 新兴方案 | 可能颠覆传统模式的创新 | |
| 边界竞品 | 部分功能重叠的周边产品 | |
```

### 第二层：第一性原理拆解

```
竞品 = 核心机制 + 解决的问题 + 放弃的权衡
```

```markdown
| 产品 | 核心机制/公式 | 解决的核心问题 | 放弃的权衡 |
|------|--------------|---------------|-----------|
| 竞品A | | | |
| 竞品B | | | |
```

### 第三层：维度对比矩阵

**根据产品类型选择对比维度：**

**SaaS 产品：**
| 维度 | 说明 |
|------|------|
| 功能覆盖 | 核心功能、高级功能、集成能力 |
| 定价模式 | 订阅制、按用量、企业定制 |
| 用户规模 | 用户数、增长率 |
| 技术架构 | 云原生、本地部署、开源 |

**DApp/DeFi：**
| 维度 | 说明 |
|------|------|
| TVL | 总锁仓量 |
| 交易量 | 24h/7d 交易量 |
| APY | 收益率 |
| Gas 费用 | 交易成本 |

**通用维度：**
| 维度 | 说明 |
|------|------|
| 上手难度 | 学习曲线 |
| 资金/时间门槛 | 启动成本 |
| 用户体验 | UI/UX 评分 |

        │       复杂度                     │
        │  低 ←───────────────→ 高        │

## Report Structure（报告结构）

```markdown

- 分析目标
- 核心问题

## 2. 竞品范围
| 类型 | 产品 | 市场地位 |

## 3. 第一性原理拆解
| 产品 | 核心机制 | 解决问题 | 权衡 |

## 4. 核心维度对比
（根据产品类型的对比表格）

## 5. 架构/流程可视化
（ASCII 图展示关键差异）

## 6. SWOT 分析
（主要竞品的 SWOT）

## 7. 差异化机会矩阵
（机会矩阵图）

## 8. MVP 切入点建议
（P0/P1/P2 优先级方案）

## 9. 技术选型建议（如适用）
| 需求 | 推荐方案 | 理由 |

## 10. 总结与行动项
- 核心洞察
- MVP 优先级
- 下一步行动
```

## Output Format Requirements

**模式控制：**

| 模式 | 输出长度 | 重点 |
|------|---------|------|
| `scan` | ~1000-2000 字 | 快速对比 + 机会识别 |
| `deep` | ~3000-5000 字 | 完整分析 + 可视化 + 建议 |

**必需包含：**
1. 竞品分类表
2. 第一性原理拆解表
3. 维度对比矩阵
4. 至少一个 ASCII 可视化
5. 主要竞品的 SWOT
6. 差异化机会矩阵
7. 优先级分类的切入点建议

## Common Mistakes

| 错误 | 正确做法 |
|------|----------|
| 没有区分 scan/deep 模式 | 根据 depth 参数控制输出长度 |
| 只列功能不解释 | 必须说明"为什么这样设计" |
| 缺少可视化 | 至少包含一个 ASCII 图 |
| SWOT 格式不统一 | 使用标准四象限格式 |
| 建议没有优先级 | 所有切入点必须标注 P0/P1/P2 |
| 信息来源单一 | AI 内部知识 + 联网搜索 + 用户补充 |
